# 05. Bus (`internal/bus`)

## 1. Назначение

Единая шина событий, связывающая стадии пайплайна. Позволяет запускать систему монолитом
(in-memory) или распределённо (Redis), отделяя `collector` от `classificator` и `dispatcher`
для независимого масштабирования в k8s.

## 2. Ответственность и границы

**Делает:**
* Публикация/подписка на типизированные топики (`updates.new`, `updates.classified`, ...).
* Гарантия доставки at-least-once (в Redis-режиме: ack, повторная доставка, DLQ).
* Сериализация события + перенос метаданных обогащения (значения из `context.Context`
  явным образом упаковываются в конверт сообщения — контекст не сериализуется).

**НЕ делает:**
* Не содержит бизнес-логики маршрутизации (это `orchestrator`).
* Не занимается дедупликацией (at-least-once компенсируется идемпотентностью потребителей).

## 3. Публичный интерфейс

```go
type Bus interface {
	Publish(ctx context.Context, topic string, msg Message) error
	// Subscribe блокируется до отмены ctx; handler должен быть идемпотентен.
	Subscribe(ctx context.Context, topic, group string, handler Handler) error
}

type Message struct {
	ID       string            // uuid, для трассировки
	Version  int               // версия схемы конверта (см. §9)
	Event    UpdateEvent       // общий тип из internal/model (см. 03-parser.md §9)
	Metadata map[string]string // обогащение (feed_id, verdict, trace_id, ...)
}

type Handler func(ctx context.Context, msg Message) error
```

Ошибка из `Handler` = nack: сообщение будет доставлено повторно (Redis-режим) или залогировано (in-memory).

## 4. Внутреннее устройство

### Реализации

| Реализация | Механизм | Использование |
|-----------|----------|----------------|
| `memorybus` | буферизованные каналы Go | монолит, тесты |
| `redisbus` | Redis Streams + consumer groups (`XADD`/`XREADGROUP`/`XACK`) | распределённый режим |

### Redis-режим
* Каждый топик — отдельный stream; `group` — consumer group (масштабирование потребителей).
* `XAUTOCLAIM` для перехвата зависших сообщений умерших потребителей.
* После N неудачных доставок сообщение уходит в DLQ-стрим `<topic>.dlq`.
* Сериализация: JSON (просто и отлаживаемо; переход на msgpack — при необходимости).

### Обогащение через context
Внутри одного процесса обогащение передаётся через `context.Context`; на границе шины
значения явно копируются в `Message.Metadata` и восстанавливаются в контекст на стороне потребителя.

## 5. Зависимости

* `github.com/redis/go-redis/v9` (только `redisbus`).

## 6. Конфигурация

```yaml
bus:
  driver: memory        # memory | redis
  redis:
    addr: localhost:6379
    stream_max_len: 10000
    max_deliveries: 5   # после — в DLQ
    block_timeout: 5s
```

## 7. Ошибки и крайние случаи

* Недоступен Redis — `Publish` возвращает ошибку с retry на вызывающей стороне; подписчики переподключаются с backoff.
* Переполнение стрима — `MAXLEN ~` обрезает старые сообщения (метрика потерь обязательна).
* Повторная доставка — норма (at-least-once); потребители обязаны быть идемпотентны.
* Сообщение-«отрава» (постоянно падающий handler) — уходит в DLQ, алерт оператору.

## 8. Тестирование

* Контрактный тест-набор, прогоняемый на обеих реализациях (memory и redis через testcontainers/miniredis).
* Сценарии: доставка, ack, повторная доставка после сбоя потребителя, DLQ, конкурентные потребители в группе.

## 9. Открытые вопросы и принятые решения

* **Список топиков и версионирование — решено (базовый набор + версия в конверте)**:
  топика три — по границам стадий пайплайна:
  * `updates.new` — новое событие после дедупликации (collector-часть → worker);
  * `updates.classified` — событие с вердиктом «важно» (worker → dispatcher);
  * `<topic>.dlq` — DLQ каждого топика (см. §4).
  Список — константы пакета `bus` (не строки по месту вызова). Версионирование —
  поле `Version` в конверте `Message` (не в имени топика): потребитель читает
  старые и новые версии (толерантный JSON-unmarshal, новые поля — только добавляются);
  топики вида `updates.new.v2` заводятся только при несовместимом breaking-изменении
  (актуально лишь в распределённом режиме с постепенным деплоем, фаза 5).
* **Retry-топик для LLM — решено (не нужен)**: при недоступности LLM приложение
  падает (fail fast, см. [08-llm.md](08-llm.md) §9) — отложенные повторы внутри шины
  бессмысленны; после рестарта события обрабатываются заново (идемпотентность
  через fingerprint и `dispatches`).
