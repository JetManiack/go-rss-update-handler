# 09. Prompt (`internal/prompt`)

## 1. Назначение

Управление промптами для LLM. Промпты — это `.md`-файлы с Go-шаблонами (`text/template`).
Встроенные промпты поставляются в бинаре через `go:embed`; пользователь может переопределить
любой из них файлами в своей директории — **пользовательский файл имеет приоритет над встроенным**.

## 2. Ответственность и границы

**Делает:**
* Загрузка промптов: сначала пользовательская директория, затем fallback на `go:embed`.
* Рендеринг шаблонов с типизированными данными (`text/template`).
* Валидация шаблонов при старте (все встроенные + переопределённые должны компилироваться).
* **Hot-reload (принято):** отслеживает изменения пользовательской директории (fsnotify)
  и перезагружает промпты без перезапуска процесса (см. §4).

**НЕ делает:**
* Не вызывает LLM и не знает семантики промптов.
* Не редактирует промпты сам — только читает и рендерит.

## 3. Публичный интерфейс

```go
type Registry interface {
	// Render рендерит промпт по имени (например, "classify") с данными data.
	Render(name string, data any) (string, error)
}

func New(userDir string) (Registry, error) // userDir может быть пустым
```

## 4. Внутреннее устройство

### Структура файлов

```
internal/prompt/
├── prompt.go
├── registry.go
└── builtin/            # встраивается через go:embed
    ├── classify.md         # основной промпт классификации
    └── classify_system.md  # системная инструкция
```

* `//go:embed builtin/*.md` → `embed.FS`.
* Резолвинг имени: `<userDir>/<name>.md` существует → используется он; иначе `builtin/<name>.md`.
* Все шаблоны парсятся один раз при создании `Registry`; ошибки — фатальны на старте
  (fail fast, а не при первом событии).
* **Hot-reload (принято):** `fsnotify` на `userDir`; при изменении файла шаблон
  перепарсивается в стороне и атомарно подменяется (`atomic.Pointer` на набор шаблонов).
  Битый новый шаблон не применяется: остаётся предыдущая версия + `error`-лог и метрика.
* Данные для `classify.md`: текущее обновление, срез истории важных обновлений, метаданные фида.

### Хедер промпта и версионирование (принятое решение)

Каждый файл промпта начинается с **YAML-хедера** (front matter, отделяется `---`):

```yaml
name: classify
version: "1.0.0"
critical: true
description: "Classify an update (...)"
```

* `name` — имя промпта, должно совпадать с именем файла (`classify.md`) — иначе ошибка валидации.
* `version` — семантическая версия (semver); именно она пишется в LLM-трейсы
  (`prompt_version`, см. [12-observability.md](12-observability.md)) и используется в eval (фаза 7).
* `critical` — промпт обязателен для работы пайплайна: отсутствие/битый файл — fail fast
  на старте; для некритичных — warning и fallback на встроенный.
* `description` — назначение промпта (документация, в рендер не попадает).

Хедер отрезается при загрузке — в LLM уходит только тело шаблона. Отсутствие хедера
или обязательных полей (`name`, `version`) — ошибка валидации на старте/hot-reload.
Переопределяя встроенный промпт, пользователь обязан указать собственную версию —
так в трейсах видно, какой именно вариант промпта дал вердикт.

### Пример шаблона (`classify.md`)

```markdown
---
name: classify
version: "1.0.0"
critical: true
description: "Classify an update against the feed's recent important updates"
---
Оцени важность обновления проекта.

## Текущее обновление
{{ .Current.RawContent }}

## Последние важные обновления (для сравнения)
{{ range .History }}- {{ .Event.PublishedAt }}: {{ .Verdict.Reason }}
{{ else }}Истории важных обновлений нет.
{{ end }}

Ответь строго JSON: {"important": bool, "category": string, "confidence": float, "reason": string}
```

## 5. Зависимости

* stdlib: `embed`, `text/template`, `io/fs`.
* `github.com/fsnotify/fsnotify` — hot-reload пользовательских промптов.
* YAML-парсер (тот же, что в `internal/config`) — разбор хедера промпта.

## 6. Конфигурация

```yaml
prompt:
  user_dir: /etc/gruh/prompts   # пустое значение = только встроенные
  hot_reload: true              # fsnotify на user_dir (принятое решение)
```

## 7. Ошибки и крайние случаи

* Битый пользовательский шаблон — ошибка на старте с именем файла и позицией (не тихий fallback на встроенный, чтобы пользователь заметил).
* Отсутствующий/невалидный YAML-хедер, несовпадение `name` с именем файла,
  не-semver `version` — ошибка валидации (для `critical: true` — fail fast).
* Пользовательский файл ссылается на несуществующее поле — ошибка рендера наверх (classificator пометит событие failed).
* `user_dir` не существует — не ошибка, работаем на встроенных, warning в лог.
* Экранирование не требуется (`text/template`, не `html/template`) — промпт это plain text.

## 8. Тестирование

* Unit: приоритет пользовательских файлов над встроенными, fallback, ошибки парсинга/рендера.
* Unit: разбор YAML-хедера (валидный/отсутствующий/битый, несовпадение `name`,
  не-semver `version`), отрезание хедера от тела перед рендером.
* Golden-тесты рендера `classify.md` на фиксированных данных (защита от случайных правок промпта).

## 9. Открытые вопросы и принятые решения

* **Hot reload — решено (да)**: fsnotify на `user_dir`, атомарная подмена набора
  шаблонов, битый шаблон не применяется (см. §4).
* **Версионирование промптов — решено (YAML-хедер)**: каждый файл начинается
  с front matter (`name`, `version` (semver), `critical`, `description`, см. §4);
  в `prompt_version` LLM-трейсов пишется `version` из хедера вместо хэша содержимого.
